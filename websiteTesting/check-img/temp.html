HELO Restriction Goodness

Posted by SysAdmin 21 November 2011 at 15h11

I recently had a DNS breakage issue with a hosting providers upstream name server, causing all lookups to fail for a short while on my web/mail host (where this blog runs incidentally). And, while I do host some DNS zones here to and point resolv.conf to localhost, I was also forwarding all outoing DNS requests to the provider's single DNS server. Bad idea. Obviously. I now forward out to Google's public servers, and so far so good.

In any event, a knock on effect of the DNS shenanigans was a mailq build-up because gmail's MX's couldn't be resolved and all my incoming mail which normally ultimately forwards to my gmail address, got stuck.

I noticed however that a number of queued messages were also spam. Now, while gmail does an amaznig job of filtering out spam, so it'd be unlikely I'd ever see these I decided that the good netizen thing to do is drop more of these spam message if possible up front and save my server processing and forwarding them on as well as saving Google from ever seeing them.

I didn't have the time today to re-implement the greylisting I had before a server rebuild some time ago, nor does my little VPS have the resources to run something like Spamassassin. So I simply added a few basic restrictions from the Postfix documentation, not expecting much to change.

I was very pleasantly surprised to immediately see a bunch of incoming spam rejected by none other than a simple check on the remote server's HELO hostname. Nice!

So I highly reccomend anyone that runs postfix as an MTA, particularly if it's your MX to add this to your main.cf:

# Don't talk to mail systems that don't know their own hostname. smtpd_helo_restrictions = reject_unknown_helo_hostname

Windows 7 IP address dumbness

Posted by SysAdmin 14 November 2011 at 07h53

I need to do a post soon(-ish) about my new toy: One of these (an HP MicroServer). However, in the meantime let's just give a mention to yet another annoying restriction of Microsoft's latest desktop OS...

On any other OS (read Linux, but I'm pretty sure OSX too - although I suppose I should test it) you can have not only 2 (or more) different static IPs aliased to one physical network device, but also easily have the primary IP be DHCP and also add 1 or more static aliased IPs too. However it seems that in Windows 7 land this is not an easy task. You can easily add additional IPs if your primary is static, but that's it. If you assign a DHCP IP to an interface, the dialog to add addtional IPs gets grayed out! Arg!

Apparently it may be possible to get around this with a registry edit. I'll let you know as soon as I've completed my Ubuntu VM installation :)
